## 2 версии сервиса

- Запускаются как 2 deployment
- Версия определяется лэйблом `version: v1 | v2`
	По лейблу будет выполняться маршрутизация

## Маршрутизация

- Добавлен `DestinationRule` с двумя subset:
	- `v1` по `version: v1`
	- `v2` по `version: v2`
- Реализован `fallback` через `outlierDetection`
	```
	outlierDetection:
	  consecutive5xxErrors: 5
	  interval: 1s
	  baseEjectionTime: 5s
	  maxEjectionPercent: 50
	```
	В случае 5 ошибок в течении секунды рут будет выкинут на 5 секунд. Так может выпасть 1 из 2 маршрутов. Таким образом при отказе одного из подов запросы начнет принимать второй
-  Circuit Breaking реализован через `connectionPool`
	```
	connectionPool:
	  tcp:
	    maxConnections: 5
	  http:
	    http1MaxPendingRequests: 100
	    maxRequestsPerConnection: 10
	```
	Может быть до 5 соединений с 10 запросами на соединение. В очереди на отправку по этим соединениям мб не более 100 сообщений
	
- Добавлен VirtualService
	Матчится на uri `/ping` и роутит на `subset`'ы v1/v2 в соотношении `90:10`
- Механизм повторов реализован через retries 
	```
	retries:
	  attempts: 3
	  perTryTimeout: 1s
	  retryOn: 5xx,connect-failure,reset,refused-stream
	```
	До трех попыток, по секунде на каждую, повторная попытка в случае одной из ошибок. По истечении попыток будет применен `outlierDetection`(см выше)

## Фича флаги

- Добавлен `envoy-filter`
- Применяется к шлюзу(возникли проблемы при работе через `kubectl port-forward svc/booking-service 8080:80`) - пришлось добавить шлюз
- Добавляет первым (`INSERT_FIRST`) правило маршрутизации, которое для запросов на `/ping` с `x-feature-enabled=true` отправляет на кластер для v2

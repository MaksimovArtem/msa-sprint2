# ADR-002: Стратегия миграции данных и целевая архитектура Booking Service

Автор: Максимов Артем

## 1. Контекст

Hotelio - монолитная система на Java/Spring, содержащая в себе 5 модулей для бронирования
В рамках расширения, решено переводить его на микросервисную архитектуру как описано в **ADR-001: Миграция монолита на микросервисы** (задание 1)

Параметры системы-монолита:
- Использование REST для синхронных вызовов сервисов
- Общая база данных для всех компонентов монолита

Первым(по заданию 2) выносится Booking Service

## 2. Требования

В рамках перехода системы Hotelio к микросервисной архитектуре выделяются:
- отдельный `booking-service`, отвечающий за управление бронированиями
- отдельный `booking-history-service`, хранящий информацию о бронированиях

При выделении новых сервисов необходимо:
- обеспечить перенос существующих данных бронирований в `booking-service`
- обеспечить передачу этих данных в `booking-history-service` через события Kafka
- сохранить консистентность между новым сервисом и монолитом

Параметры системы после перехода:
- Использование REST для синхронных вызовов сервисов, остающихся в монолите
- Использование gRPC для взаимодействия монолита с новым `booking-service`
	- Использование конкретного booking.proto
- Использование Kafka для асинхронного обмена событиями
- Отдельные базы данных для новых сервисов

## 3. Решение

### 3.1 Миграция данных

На первом запуске booking-service выполняется bootstrap-миграция.
Сервис получает существующие бронирования из монолита(по REST API запроса `/api/bookings`)

>[!note] 
 p-o-y-1.0.0.jar файл в задании, к сожалению, сломан и выдает ошибку на запрос `/api/bookings` несмотря на логику и то, что в задании явно сказано, что этот запрос не менялся

сохраняет их в свою БД и публикует события `BookingCreated` в Kafka.
### 3.2 Архитектура TO BE

Монолит остаётся основной системой, но операция создания бронирования больше не выполняется внутри него. Вместо этого монолит выступает как клиент и перенаправляет запрос на создание бронирования в новый микросервис `booking-service` через gRPC.

#### Поток создания бронирования

1. Пользователь отправляет запрос на создание бронирования в монолит.
2. Монолит принимает запрос и вызывает метод `CreateBooking` у `booking-service` через gRPC.
3. `booking-service` выполняет всю бизнес-логику, связанную с созданием бронирования(валидации пользователя и отеля на основе REST запросов).
4. Сервис сохраняет бронирование в своей собственной базе данных.
5. После успешного сохранения сервис публикует событие `BookingCreated` в Kafka.

Таким образом `booking-service` становится точкой входа для всей логики создания бронирований.
#### Асинхронная обработка истории бронирований

Для формирования статистики создаётся отдельный сервис `booking-history-service`.

Этот сервис:
- подписывается на Kafka
- слушает события `BookingCreated`
- обрабатывает полученные события
- сохраняет агрегированные данные или статистику в собственной базе данных

`booking-history-service` не обращается напрямую к базе `booking-service`, а получает информацию только через события Kafka. 
Это позволяет формировать статистику и аналитические данные(в текущем решении по сути копипаста, тк требований по изменению/агрегации данных не было) без дополнительной нагрузки на основной сервис бронирований.
## 4. Альтернативы

### 4.1 Не делать миграцию данных

В этом случае данные, накопленные до миграции, осталась бы в монолите, что усложнило бы аналитику, ради которой создавался `booking-history-service`

# ADR-001: Миграция монолита на микросервисы

 <a name="_hjk0fkfyohdk"></a>**Автор: Максимов Артем**
## 1. Контекст

### 1.1 Описание задачи

Необходимо проработать миграцию монолита сервиса Hotelio на микросервисную архитектуру
Монолит написан на Java и предоставляет REST API для пользователей
### 1.2 Текущая архитектура

Монолит содержит несколько бизнес-модулей:
- BookingService
- UserService
- HotelService
- PromoCodeService
- ReviewService

Все модули:
- работают в одном приложении
- используют единую базу данных(Posgres)
- вызывают друг друга напрямую через методы/интерфейсы

BookingService выполняет роль оркестратора - от него зависят все остальные модули
### 1.3 Проблемы текущей архитектуры

Уже описанные проблемы:

1. **Сложность сопровождения**
    - Любое изменение требует понимания всей кодовой базы.
    - Невозможно менять один модуль, не затрагивая другие.
2. **Низкая масштабируемость**
    - При пиковых нагрузках (например, на бронирование) нельзя масштабировать только нужный компонент.
    - Производительность неравномерна.
3. **Невозможность гибкой разработки**
    - Разные команды не могут работать независимо.
    - Трудно внедрить быстрый CI/CD.
4. **Ограничения на фронтенде**
    - REST API монолита слишком универсален и плохо адаптируется под разные платформы.
    - Нет поддержки BFF.
5. **Сложности с запуском новых фич**
    - Большой риск ошибок из-за плотной связанности.
    - Тестирование требует понимания всех зависимостей.

Дополнительно обнаруженные проблемы:
1. **Общая база данных**
	- Изменения в схемах затронут базу, которая используется всеми
2. **Синхронные вызовы модулей**
	- При проблемах с одним из модулей встанет вся система, так как взаимодействие синхронно

## 2. Требования

Необходима миграция на микросервисную архитектуру. Миграция выполняется по паттерну Strangler Fig, так что сервисы необходимо мигрировать по одному
## 3. Решение

### 3.1 Целевая архитектура
Целевая архитектура представлена в C2.puml. Диаграмма уровня С1 опущена за ненадобностью. В описанную в задании архитектуру можно было бы добавить внешние системы для оплаты или нотификации пользователей, но это не входит в задание

Фронтэнд взаимодействует с API Gateway, который маршрутизирует запросы на каждый из 5 сервисов. Сейчас в API есть явный запрос только к BookingService, но после разбиения точно появятся запросы на добавление данных и для других сервисов(наверняка API добавления отеля должно быть и в монолите, но оно не описано для упрощения задания). Каждый сервис пишет данные в свою базу.

UserService, HotelService, PromoCodeService, ReviewService генерируют события, которые позволят BookingService проводить валидацию бронирования без лишних синхронных запросов. Примеры событий: "Добавлен отель Х", "Отель Y стал недоверенным", "Пользователь Z теперь VIP"

Сервис бронирования пользуется кэшом для валидации брони, однако в случае отсутствия данных может и сделать запрос к сервисам(на схеме сознательно не отмечено, чтобы ее не захламлять)

### 3.2 Решение текущих проблем
Проблемы описаны в 1.3 текущего документа

1. **Сложность сопровождения**
    Выделение в микросервисы сделает блоки кода независимыми. При сохранении внешнего API реализацию отдельных сервисов можно будет менять независимо от других. 
2. **Низкая масштабируемость**
    Выделение в микросервисы позволит масштабировать только нагруженные сервисы
3. **Невозможность гибкой разработки**
    Выделение в микросервисы позволит работать параллельно разным командам(и даже на разных языках)
4. **Ограничения на фронтенде**
    Слой BFF улучшит взаимодействие фронтэнда с бэком
5. **Сложности с запуском новых фич**
    Уменьшение связанности компонентов за счет выделения в микросервисы
6. **Общая база данных**
	Каждый микросервис будет иметь свою базу данных
7. **Синхронные вызовы модулей**
	Возможность избежать некоторых синхронных вызовов за счет кэширования данных, полученных из событий
### 3.3 Первый выносимый сервис
Первым будет вынесен PromoCodeService

Причины:
1. Низкое влияние на систему
   При отказе сервиса в результате ошибки миграции не повлияет на основной бизнес-процесс - бронирование отеля, так как просто не применится скидка
2. Слабые зависимости от других компонентов
   Используется BookingService для проверки промокодов, сам вызывает UserService. В обоих случаях речь о GET запросах.
   ===
   Комментарий ревьюверу:
   Простота API(отсутствие POST/DELETE запросов для отзывов) кажется странным, но видимо это часть упрощения задания. В реальной системе они 100% будут - не будет же админ руками промокоды в базу вставлять, но сами запросы не выглядят сложными - обычный CRUD. В ответе используется та информация, которая есть в задании, без додумывания
   ===
3. Изолированная бизнес-логика
	- валидность промокода
	- применимость промокода к пользователю

	Не считая запроса к пользователю, логика изолирована от остальной системы. В случае использования событийной системы и кэширования на стороне PromoCodeService запрос не понадобится
### 3.4 План миграции сервисов
1) Вынесение PromoCodeService в отдельный микросервис
   Монолит обращается напрямую по HTTP
2) Добавление API Gateway
   Для взаимодействия BookingService с переносимыми сервисами
3) Вынесение ReviewService в отдельный микросервис
4) Вынесение HotelService в отдельный микросервис
5) Вынесение UserService в отдельный микросервис
6) Вынесение BookingService в отдельный микросервис
## <a name="_bjrr7veeh80c"></a>**4. Альтернативы**

### 4.1 Альтернативные сервисы для вынесения в первую очередь

#### ReviewService

Имеет даже более простой API, чем PromoCodeService и с него можно было бы начать, но в контексте BookingService он является ключевым. Без него(в случае ошибки при выделении) встанет основной бизнес-процесс и бронирование будет недоступно, так как отель будет считаться недоверенным. С учетом того, что по заданию "старые сервисы не дорабатываются" и нельзя поменять код BookingService, чтобы при ошибке валидации он пропускал проверку, а не прекращал бронирование, то вынесение этого сервиса первым неверно. Отклонено

#### Остальные сервисы

Имеют ту же проблему, что и ReviewService. Только их API и взаимодействие с другими компонентами системы еще сложнее, чем у сервиса отзывов. Отклонено
